# Smart Canvas Planner Code Guide

This project is a Streamlit app that connects to Canvas, pulls upcoming assignments
and calendar events, estimates how long assignments will take, and creates study
blocks that can be pushed back to the user's Canvas calendar.

The main idea is:

1. Fetch assignment and calendar data from Canvas.
2. Let the user edit that data before scheduling.
3. Estimate assignment effort with either default logic or an optional NRP LLM call.
4. Find real free time around Canvas events, custom blocked times, and already-planned
   study blocks.
5. Generate study blocks and optionally add them to Canvas.

## Top-Level Files

### `app.py`

This is the main controller for the Streamlit app.

It is responsible for:

- Setting up Streamlit session state.
- Showing the sidebar controls for Canvas authentication.
- Showing the NRP LLM duration-estimation settings.
- Fetching Canvas assignments and calendar events.
- Converting Canvas data into the simplified format the scheduler expects.
- Expanding custom blocked-time rows into busy calendar intervals.
- Running assignment duration estimation.
- Calling `SmartScheduler` to generate study blocks.
- Adding generated study blocks to Canvas.
- Deleting generated study blocks from Canvas.

Important functions:

- `_build_scheduler_inputs(raw_data)`: converts edited Canvas data into scheduler input.
- `_estimate_assignment_durations(assignments)`: applies default or LLM-based duration estimates.
- `run_optimizer()`: runs the full scheduling flow.
- `push_study_blocks_to_canvas()`: creates Canvas calendar events for generated blocks.
- `delete_study_blocks_from_canvas()`: removes generated study blocks from Canvas.

### `Dashboard.py`

This file contains most of the Streamlit user interface.

It is responsible for:

- Showing assignment and event tables.
- Letting the user edit assignment fields before scheduling.
- Letting the user edit assignment instructions/descriptions.
- Letting the user mark whether an assignment can be split into multiple work blocks.
- Showing generated study blocks.
- Showing assignments that could not be fully scheduled.
- Providing scheduler settings in the Settings tab.
- Providing custom blocked-time settings, like work, classes, or no-study hours.

Important UI areas:

- `Current Tasks`: editable assignment and event data.
- `Smart Schedule`: generated study blocks and Canvas sync buttons.
- `Settings`: study-hour settings, block sizing, buffers, part spacing, and blocked times.

### `requirements.txt`

Lists the Python packages needed to run the app.

The main dependencies are:

- `streamlit`: builds the web app UI.
- `canvasapi`: talks to Canvas.
- `requests`: sends HTTP requests to the LLM API.
- `pandas`: supports editable tables and data display.

## `data/` Folder

The `data` folder contains code that talks to outside data sources. Right now, that
mainly means Canvas.

### `data/canvas_client.py`

This file wraps the Canvas API so the rest of the app does not need to know the
details of Canvas API calls.

It is responsible for:

- Connecting to Canvas with the user's Canvas access token.
- Getting the current Canvas user.
- Getting favorite Canvas calendars/courses.
- Fetching Canvas assignment calendar items.
- Fetching normal calendar events.
- Looking up full assignment details from Canvas.
- Pulling assignment points, due dates, group/category weights, submission types,
  rubric info, peer-review info, availability windows, and instructions.
- Converting Canvas HTML descriptions into plain text instructions.
- Creating Canvas calendar events for study blocks.
- Deleting generated study blocks from Canvas.

Important class:

- `CanvasInterface`: the main Canvas wrapper used by `app.py`.

Important methods:

- `get_calendar_sources()`: returns calendars/courses the user can sync.
- `get_student_workload(...)`: fetches assignments.
- `get_existing_events(...)`: fetches busy calendar events.
- `push_study_block(...)`: creates a study block event in Canvas.
- `delete_generated_study_blocks(...)`: removes study blocks generated by this app.

## `logic/` Folder

The `logic` folder contains the app's scheduling and prediction logic. These files
try to avoid UI code so they can be tested and improved separately.

### `logic/duration_estimator.py`

This file estimates how long an assignment will take.

It supports two paths:

- A default heuristic estimate that runs locally.
- An optional LLM estimate using an OpenAI-compatible NRP endpoint.

The local heuristic looks at signals like:

- Assignment title/instruction cues.
- Points adjusted by gradebook category weight.
- Submission type, such as quiz, upload, discussion, or annotation.
- File extensions, such as code files, documents, slides, PDFs, or notebooks.
- Peer review and group-work fields.

It intentionally does not use instruction word count as a duration signal because
longer instructions do not always mean longer work.

Important items:

- `heuristic_assignment_duration(assignment)`: default local estimate.
- `LLMDurationEstimator`: sends assignment data to the NRP LLM API.
- `assignment_estimate_cache_key(assignment)`: avoids repeating the same LLM call.
- `NRP_CHAT_MODEL_OPTIONS`: model dropdown options for the UI.
- `NRP_ENDPOINT_OPTIONS`: endpoint dropdown options for the UI.

### `logic/scheduler.py`

This file decides when study blocks should happen.

It is responsible for:

- Determining whether an assignment can be split into parts.
- Creating available study windows from study hours.
- Removing Canvas events and user-defined blocked times from availability.
- Reserving generated study blocks so later blocks do not overlap them.
- Splitting larger assignments into multiple blocks when allowed.
- Keeping non-splittable assignments, like quizzes, in one continuous block.
- Reporting assignments that could not be fully scheduled.

Important items:

- `infer_can_split(assignment)`: guesses if an assignment can be split.
- `SmartScheduler`: the main scheduling class.
- `priority_score(assignment)`: ranks assignments by urgency, weighted importance,
  and estimated effort.
- `generate_predictions()`: returns the generated study blocks.

## Key Concepts

### Assignment Duration Estimate

The duration estimate is the app's best guess for how many minutes of active work an
assignment requires. The estimate can come from local logic or the NRP LLM.

The scheduler does not ask the LLM where to place blocks. The LLM only estimates
work time. Calendar placement is handled by deterministic scheduler logic.

### Study Block

A study block is a calendar event generated by the app. It has a start time, end
time, assignment name, course name, and scheduling details.

### Busy Event

A busy event is any time when the scheduler should avoid placing study blocks.
Busy events include:

- Canvas calendar events.
- User-defined blocked times.
- Previously generated study blocks during the current scheduling run.

### Custom Blocked Time

Custom blocked time lets the user mark recurring unavailable time, such as work,
classes, commuting, meals, or sleep.

Examples:

```text
Name: Work
Day: Weekdays
Start: 09:00
End: 17:00
```

```text
Name: Class
Day: Tuesday
Start: 13:00
End: 15:15
```

### Can Split

`Can Split` tells the scheduler whether the assignment can be broken into multiple
study blocks.

Examples:

- A project or essay can usually be split.
- A timed quiz usually cannot be split.
- The app guesses this from Canvas data, but the user can edit it.

## How To Run

From the project root:

```bash
.venv/bin/streamlit run app.py
```

Then open the local Streamlit URL shown in the terminal, usually:

```text
http://localhost:8501
```

## Things To Know Before Editing

- `app.py` coordinates the app. If you are wiring features together, start there.
- `Dashboard.py` is UI-heavy. If you are changing what the user can edit or see,
  start there.
- `data/canvas_client.py` is Canvas-specific. If you need more Canvas fields, add
  them there and pass them through `app.py`.
- `logic/duration_estimator.py` should only estimate effort, not schedule time.
- `logic/scheduler.py` should handle calendar math, free windows, blocked time,
  splitting, and prioritization.
